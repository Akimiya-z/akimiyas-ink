如果我每做完一个项目，都再从头手写一篇教程博客，这件事很快就会拖慢节奏。现在我把这一步收成了两个 skill：一个负责把真实项目过程整理成教程初稿，另一个负责审查和整理成发布版。它们不负责“替我做项目”，只负责在项目完成之后，把过程沉淀成一篇可以发到博客上的文章。

## 核心概念

- **`project-tutorial-generator`** - 把一次已经完成的代码项目过程整理成教程博客初稿，并顺手留下一份过程摘要。
- **`review-project-tutorial`** - 审查已经生成的项目教程博客，检查结构、可执行性、事实准确性，再按需要整理成发布版。
- **`tutorial-sources`** - 存放每篇教程对应的摘要文件；以后另开新对话时，可以靠它把上下文捞回来。

## 你需要准备的内容

- 这两个 skill 已经装进你的 Codex 环境
- 一个已经做完或基本做完的代码项目
- 你的博客仓库已经能正常写文章并构建
- 愿意先让文章成型，再决定要不要细修
- 10 到 20 分钟

## Step 1: 先把项目做完再写教程

- 这套流程适合“事后整理”，不适合一边开发一边写博客
- 先和 Codex 把项目做完，至少做到功能、页面、脚本或部署流程已经基本稳定
- 只有项目本身有了真实步骤、真实坑和真实结果，后面的教程才不会变成空话

## Step 2: 用生成器把真实过程整理成初稿

- 在当前对话里直接说你要把这次项目整理成教程博客，例如：

```text
把这次和 Codex 一起完成的项目过程整理成教程博客，先给我提纲。
```

- 如果你想显式调用，也可以直接写：

```text
Use $project-tutorial-generator 把这次和 Codex 协作完成的代码项目过程整理成教程博客，先给出提纲，再生成正文。
```

- 这个 skill 正常会先做三件事：
  - 读取当前对话里的真实步骤
  - 读取项目最终结果
  - 先给一个提纲让你确认

- 你确认后，它会按我当前博客仓库的目录约定，把文章写到：

```text
src/content/blog/<slug>/index.md
```

- 同时把过程摘要写到：

```text
tutorial-sources/<slug>.md
```

## Step 3: 先检查它写出来的是什么

- 初稿生成后，先看三件事：
  - 标题是不是对的
  - 步骤顺序是不是贴近真实过程
  - 有没有把关键决策和关键坑写进去

- 这一轮不要急着抠字眼，先确认文章方向没跑偏
- 如果主题变了，比如从“复盘自己怎么做”改成“教别人怎么复用仓库”，应该在这一步就纠正

## Step 4: 用审查器把初稿变成可发布版本

- 如果你只想看问题，不想立刻修改，可以这样说：

```text
审查刚生成的教程，只先列问题，不要先改。
```

- 如果你想让它边审边整理成发布版，可以这样说：

```text
审查刚生成的教程；如果问题不大，直接整理成发布版。
```

- 显式写法也可以：

```text
Use $review-project-tutorial 审查这篇代码项目教程博客；如果问题不大，直接整理成发布版。
```

- 它主要会检查这些东西：
  - frontmatter 是否完整
  - 步骤能不能照着做
  - 命令和路径是否准确
  - 有没有漏前提条件
  - 是否忠于真实过程

- 如果文章方向已经是对的，通常这一步主要做的是收紧措辞、补前提、删废话，而不是整篇重写

## Step 5: 用摘要文件解决“换对话后丢上下文”

- 如果你当场没有发文章，之后另开一个新对话也没关系
- 生成器在第一次写完时，已经留下一份摘要：

```text
tutorial-sources/<slug>.md
```

- 以后你可以在新对话里说：

```text
根据之前那篇教程的摘要，把这次项目过程重新整理成教程博客。
```

- 这样即使原始对话不在了，仍然可以靠摘要和项目现状把文章重建出来

## Step 6: 构建、检查并发布

- 文章整理完后，回到博客仓库里跑一次构建：

```bash
npm run build
```

- 只要构建通过，再正常提交和推送：

```bash
git add .
git commit -m "feat: publish tutorial blog"
git push
```

- 如果你的站点接了 Vercel、Netlify 或其他自动部署服务，推送后就会自动上线

## 补充说明

- 这两个 skill 只适合“代码项目教程博客”这个场景，不适合统计学笔记、普通随笔或 README
- 生成器负责把过程写成文章，审查器负责把文章变得更稳，不要让一个 skill 既干活又包办所有决策
- 如果你平时更习惯自然说话，不一定每次都要显式写 `$project-tutorial-generator` 或 `$review-project-tutorial`；但在触发不稳的时候，显式写法最省事
- 真正决定文章质量的，不是 prompt 写得多花，而是你有没有真实做完这个项目、有没有留下真实决策和真实坑
- 这篇文章里的输出路径是按我当前博客仓库的约定写的；如果你以后换博客结构，应该先改对应的 skill 配置或输出目录

## 下一步

- 挑一个你刚和 Codex 做完的项目，按这套流程发第一篇教程博客
- 如果你想让别人也复用这套工作流，可以把可移植版 skill 仓库一起公开出来
- 等你积累出 3 到 5 篇项目教程后，再考虑把文章列表页或标签页专门为“教程博客”做一点整理

## 故障排除

- **问题**：它开始把文章写成普通博客或学习笔记  
  **处理**：明确说这是“代码项目教程博客”，并指出主题是某个已经完成的项目过程，不是随笔或笔记。

- **问题**：新对话里它不知道之前发生了什么  
  **处理**：把 `tutorial-sources/<slug>.md` 对应的摘要文件路径给它，或者让它先读摘要再写正文。
